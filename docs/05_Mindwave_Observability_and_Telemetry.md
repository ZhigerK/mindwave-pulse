# ЕДИНАЯ СИСТЕМА НАБЛЮДАЕМОСТИ И ТЕЛЕМЕТРИИ (OBSERVABILITY & EVENT TRACKING)
**Продукт:** Платформа Mindwave (Гибридная образовательная экосистема)
**Стек:** Grafana Stack, ClickHouse / PostHog, Node.js, Kafka/RabbitMQ
**Статус:** Черновик

---

## 1. Цели внедрения единой системы телеметрии

1.  **Технический мониторинг (Grafana Stack):** Оперативное выявление сбоев в микросервисах, контроль нагрузки и API-запросов.
2.  **Сбор цифрового следа (Event Tracking):** Накопление исторических данных обо всех действиях студентов (клики, просмотры, паузы, сдачи тестов).
3.  **Генерация "умных" триггеров (Predictive Analytics):** Переход от жестких правил ("не был 3 дня") к динамическим ("пользователь перематывает видео, часто ставит на паузу и медленно решает тесты — риск отвала 85%, отправляем триггер ментору сейчас").

---

## 2. Архитектура System Observability (Технический слой)

Для контроля здоровья платформы мы используем стандартный стек вокруг **Grafana** (LGTM Stack):

* **Loki (Logs):** Централизованный сбор логов со всех Node.js микросервисов и Keycloak.
* **Grafana (Dashboards):** Единый интерфейс для визуализации метрик и настройки базовых алертов (например, "CPU > 80% -> Alert in Telegram").
* **Tempo (Traces):** Распределенная трассировка запросов (OpenTelemetry). Позволяет увидеть, как запрос прошел от API Gateway через Auth Service в LMS Service.
* **Mimir / Prometheus (Metrics):** Сбор метрик (количество запросов, время ответа, объем памяти).

---

## 3. Архитектура Event Tracking (Бизнес-слой и сбор действий)

Для сбора действий пользователей реляционные базы (PostgreSQL) не подходят — они быстро "задохнутся" от объема логов. Нам нужно колоночное хранилище или специализированный CDP (Customer Data Platform).

### Вариант А: Custom-решение (Kafka + ClickHouse + Grafana)
*Рекомендуется, если мы хотим полностью владеть данными и строить сложные SQL-запросы в Airflow.*

1.  **Генерация события:** Фронтенд (React/Vue) или Бэкенд (Node.js) генерирует JSON-событие (`VIDEO_PAUSED`, `TEST_STARTED`).
2.  **Брокер сообщений:** Событие отправляется в Apache Kafka (или RabbitMQ).
3.  **Хранилище (ClickHouse):** Данные из Kafka переливаются в базу данных ClickHouse. Это невероятно быстрая колоночная БД, созданная специально для аналитики огромных массивов данных.
4.  **Визуализация и Аналитика:** Grafana имеет нативную интеграцию с ClickHouse. Аналитики могут строить дашборды, а **Apache Airflow** может раз в час делать сложные SELECT-запросы в ClickHouse, чтобы найти аномалии и сгенерировать новый триггер для Ментора.

### Вариант Б: Использование готовых сервисов (Off-the-shelf)
*Рекомендуется для быстрого старта (MVP), чтобы не разворачивать ClickHouse руками.*

* **PostHog (Open-Source / SaaS):** Идеальный инструмент для нашего кейса. Он собирает все действия пользователей (автоматический автокапчур кликов + кастомные события). У него есть встроенная аналитика, воронки удержания, и главное — **Webhooks**. Постхог может сам стать источником "неопределенных триггеров", отправляя вебхук на наш Node.js бэкенд, когда пользователь совершает определенную последовательность действий.
* **Segment (SaaS) / RudderStack (Open-Source):** Маршрутизаторы событий. Фронтенд шлет событие в Segment, а Segment уже раздает его в амплитуду, базу данных, Airflow и т.д.

**Рекомендация для Mindwave:** Использовать **PostHog** для Product Analytics (быстрый старт, готовые воронки, вебхуки для триггеров) и **ClickHouse + Grafana** в будущем, когда потребуется своя ML-инфраструктура.

## 4. Стандарт схемы данных (Event Schema)

Чтобы данные были полезными для Airflow и ML в будущем, *каждое* действие пользователя должно отправляться в шину данных (или PostHog/ClickHouse) в строго стандартизированном JSON-формате.

**Обязательная структура (The Mindwave Event Standard):**

```json
{
  "event_id": "uuid-v4",
  "timestamp": "2023-10-27T10:00:00Z",
  "event_name": "lesson_video_paused", 
  "actor": {
    "user_id": "12345",
    "role": "ROLE_CHILD",
    "session_id": "abc-987"
  },
  "context": {
    "device_type": "mobile",
    "os": "iOS",
    "app_version": "1.2.0"
  },
  "payload": {
    "course_id": "math_5_grade",
    "lesson_id": "fractions_01",
    "video_timestamp_seconds": 125,
    "total_video_duration": 600
  }
}
```

**Ключевые типы событий для накопления базы:**
* `app_opened` / `app_closed` (расчет реального времени сессии).
* `lesson_started` / `lesson_completed` (воронка доходимости).
* `video_played` / `video_paused` / `video_seeked` (перемотка — индикатор скуки или непонимания).
* `test_started` / `Youtubeed` (с фиксацией времени на каждый вопрос).
* `chat_message_sent` (активность общения с ментором).

## 5. Как это превращается в "Неопределенные триггеры" (Feedback Loop)

Связка накопленных данных с нашей архитектурой (Airflow + Node.js) работает так:

1.  **Сбор (События копятся в ClickHouse / PostHog):** Платформа просто складывает туда JSON-ы каждый день.
2.  **Анализ (Data Science):** Через полгода у нас есть датасет. Аналитик видит паттерн: *«Если Adult-студент перематывает видео назад больше 3 раз за урок, а потом ставит паузу на тесте дольше 5 минут — он отменяет подписку в течение недели с вероятностью 80%».*
3.  **Создание динамического триггера (Airflow):**
    * Аналитик пишет новый DAG в Apache Airflow.
    * Airflow раз в 15 минут делает SQL-запрос к ClickHouse: *"Найди пользователей, подходящих под паттерн X за последний час"*.
    * Если находит — Airflow пушит сообщение в RabbitMQ: `CreateMentorTask { priority: "high", reason: "predictive_churn_risk" }`.
4.  **Действие (Node.js CRM):** Ментор получает задачу и пишет студенту, предотвращая отток, предсказанный на основе больших данных.
