# СТРАТЕГИЯ ВЕРСИОНИРОВАНИЯ КОНТЕНТА И CI/CD (CONTENT-AS-CODE)
**Продукт:** Платформа Mindwave (Гибридная образовательная экосистема)
**Стек:** Git (GitHub/GitLab), Google Cloud Storage (GCS), MongoDB, Node.js
**Статус:** Утверждено

---

## 1. Суть подхода "Контент как код" (Content-as-Code)

Традиционные LMS используют визуальные конструкторы (WYSIWYG), где изменения применяются мгновенно и ломают прогресс текущих студентов.

В Mindwave мы применяем инженерный подход: образовательный контент отделен от базы данных пользователей. Он пишется как код, версионируется в Git и доставляется через CI/CD пайплайн.

### Архитектурная триада хранения:
1. **Git-репозиторий (Структура и Тексты):** Хранит JSON-файлы со структурой модулей, тестами и текстовыми уроками (в формате Markdown).
2. **Google Cloud Storage (Тяжелые медиа):** Хранит видео, аудио, изображения и PDF. Файлы отдаются студентам через Google Cloud CDN для высокой скорости загрузки по всему миру.
3. **MongoDB (Быстрая отдача клиенту):** "Собранный" релиз курса, к которому обращается фронтенд (SPA/PWA) приложения для моментального отображения.

---

## 2. Использование Google Cloud Storage (GCS)

Медиафайлы не версионируются в Git (он для этого не предназначен). Для них действует правило **«Только добавление» (Append-Only)** в GCS.

* **Структура бакетов (Buckets):** `gs://mindwave-content-prod/courses/{course_code}/media/`
* **Неизменяемость:** Если методист хочет заменить видео в уроке «Дроби», он не перезаписывает файл `fractions_v1.mp4`. Он загружает `fractions_v2.mp4` в Google Cloud Storage.
* **Связка с Git:** В JSON-файле урока в Git-репозитории методист просто меняет ссылку с `.../fractions_v1.mp4` на `.../fractions_v2.mp4`. Старое видео остается в GCS, чтобы старые версии курса продолжали работать для текущих студентов.

## 3. Семантическое версионирование (SemVer) и Version Pinning

Каждый курс имеет жесткую версию в формате `vX.Y.Z` (Major.Minor.Patch). В PostgreSQL (в таблице записей `enrollments`) за студентом **фиксируется версия** (`course_version`: "v1.0.0").

### Логика обновлений для студента:

* **PATCH (v1.0.1 ➔ v1.0.2): Исправление багов.** * *Пример:* Исправлена опечатка в Markdown-тексте или заменена неработающая ссылка. Структура уроков и тестов не тронута.
    * *Реакция системы:* MongoDB просто обновляет документ текущей версии. Студент видит исправление мгновенно (Hotfix).
* **MINOR (v1.0.0 ➔ v1.1.0): Безопасное добавление.**
    * *Пример:* Добавлен новый бонусный (необязательный) урок.
    * *Реакция системы:* В MongoDB создается новая версия. Для текущих студентов система делает автоматический "мягкий" апгрейд (так как их прогресс не ломается).
* **MAJOR (v1.0.0 ➔ v2.0.0): Ломающие изменения (Breaking Changes).**
    * *Пример:* Удалены уроки, полностью переписаны тесты, изменен порядок модулей.
    * *Реакция системы:* Студент **остается на версии v1.0.0**. Система не трогает его траекторию. В интерфейсе студента (Adult) или Родителя (Parent) появляется уведомление о возможности явного перехода.

## 4. Пайплайн доставки контента (CI/CD Workflow)

Как выглядит ежедневная работа методиста и DevOps-инженеров:

1. **Создание ветки (Drafting):** Методист создает ветку `feature/update-math-test` в репозитории курса.
2. **Медиа в GCS:** Методист загружает новые картинки/видео в Google Cloud Storage и получает их публичные CDN-ссылки.
3. **Редактирование JSON/MD:** Вставляет эти ссылки в конфигурационные JSON-файлы курса и коммитит изменения.
4. **Ревью (Pull Request):** Старший методист проверяет изменения в Git (видит разницу в текстах/тестах как обычный diff кода).
5. **Релиз (Merge & Tag):** PR принимается в ветку `main`. Навешивается тег релиза (например, `v2.0.0`).
6. **Автоматизация (GitHub Actions / GitLab CI):**
    * CI-сервер ловит тег `v2.0.0`.
    * Делает POST-запрос (Webhook) на наш внутренний `LMS Service` (Node.js).
    * В запросе передает ссылку на скачивание архива репозитория.
7. **Запись в MongoDB:** `LMS Service` распаковывает архив, валидирует JSON-схемы и записывает в коллекцию `Courses` новый независимый документ `course_math_5_v2_0_0`.

## 5. Пользовательский сценарий: Явный апгрейд курса (Explicit Upgrade)

Если вышла версия `v2.0.0` (Major), студент версии `v1.0.0` видит плашку: *"Доступна новая версия курса. Хотите обновиться?"*

**Шаги алгоритма при клике на "Обновить":**
1. **Расчет потерь (Dry-Run):** Система сравнивает `v1.0.0` и `v2.0.0`. Фронтенд показывает предупреждение: *"При обновлении ваш прогресс в Модуле 3 будет сброшен, так как мы полностью переработали тесты. Вы уверены?"*
2. **Миграция:** Если пользователь согласен, Node.js сервис переносит прогресс (XP и галочки) для тех уроков, чьи `lesson_id` совпали в обеих версиях.
3. **Обновление БД:** В PostgreSQL в таблице `enrollments` поле `course_version` меняется на `v2.0.0`.
4. **Перезагрузка интерфейса:** Студент видит новую структуру курса с актуальным (пересчитанным) прогрессом.
