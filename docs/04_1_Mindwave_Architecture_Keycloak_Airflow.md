## 1. Роль Keycloak (Identity and Access Management)

Вместо того чтобы писать собственный микросервис авторизации, мы делегируем эти функции Keycloak. Он берет на себя регистрацию, логин, восстановление паролей, SSO, выдачу JWT-токенов и управление ролями (RBAC).

### 1.1. Настройка ролевой модели (Realms & Roles)
В Keycloak мы создаем один Realm (например, `mindwave-realm`) и заводим в нем роли на уровне клиента (Client Roles):
* `ROLE_ADULT` — имеет доступ к контенту и биллингу.
* `ROLE_CHILD` — имеет доступ только к контенту и геймификации.
* `ROLE_PARENT` — имеет доступ только к дашборду и биллингу.
* `ROLE_MENTOR` — имеет доступ к CRM и чатам.

### 1.2. Решение проблемы связки Parent ↔ Child
Поскольку Keycloak по умолчанию не имеет встроенной жесткой иерархии "Родитель-Ребенок", мы реализуем эту связку через **User Attributes** (Атрибуты пользователя) или **Groups**:
* При регистрации Родителя в Keycloak создается `User_Parent`.
* При создании профиля Ребенка создается `User_Child`, которому в `attributes` прописывается `parentId: [ID_Родителя]`.
* В JWT-токене Родителя (через кастомный Token Mapper) мы можем сразу «зашивать» массив `childrenIds`.
* *Альтернатива:* Keycloak занимается только AuthN/AuthZ, а связка `Parent_ID` <-> `Child_ID` хранится в нашей собственной реляционной БД (например, в сервисе User Management / Billing), чтобы легче было строить SQL-запросы.

### 1.3. Жизненный цикл запроса с Keycloak
1. Клиент (SPA/PWA) обращается к Keycloak, пользователь вводит логин/пароль.
2. Keycloak возвращает `Access Token (JWT)` и `Refresh Token`.
3. Клиент делает запрос к нашему API Gateway (например, NGINX/Kong), прикрепляя токен в заголовок `Authorization: Bearer <token>`.
4. API Gateway (или сами Node.js микросервисы) валидирует подпись JWT через публичный ключ Keycloak.
5. Микросервис читает из токена роль (например, `ROLE_CHILD`) и `userId`, и на основе этого отдает нужные данные.

## 2. Роль Apache Airflow (Оркестрация и Batch-процессы)

Apache Airflow берет на себя все **события, зависящие от времени** (cron-job'ы, ежедневные проверки, генерация отчетов). Node.js микросервисы больше не держат у себя таймеры — они предоставляют API (или слушают брокер сообщений), а Airflow ими управляет.

Airflow работает на базе DAG (Directed Acyclic Graphs) — направленных графов задач.

### 2.1. DAG 1: Ежедневный расчет рисков (Daily Churn Checker)
*Запускается каждый день в 02:00 ночи.*
* **Task 1 (Extract):** Airflow делает SQL-запрос к БД прогресса (PostgreSQL/Redis) или дергает API Node.js сервиса: *"Дай список ID пользователей, у которых не было активности > 3 дней"*.
* **Task 2 (Transform/Filter):** Разделяет полученные ID на Adult и Child (запросив роли у Keycloak или БД пользователей).
* **Task 3 (Load/Action):**
    * Для Adult: Airflow дергает API сервиса уведомлений (Email/Push).
    * Для Child: Airflow кладет событие `CreateMentorTask` в RabbitMQ. Node.js CRM-сервис ментора ловит его и создает задачу в интерфейсе.

### 2.2. DAG 2: Еженедельная отчетность для Родителей (Weekly Parent Digest)
*Запускается каждую субботу в 10:00 утра.*
* **Task 1:** Сбор всех `Child_ID` и их успехов (XP, пройденные уроки) за прошедшие 7 дней из LMS БД.
* **Task 2:** Сопоставление `Child_ID` с `Parent_ID` (маппинг).
* **Task 3:** Формирование JSON-отчета и отправка его в Node.js Notification Service, который рассылает сообщения в Telegram/WhatsApp родителям.

### 2.3. DAG 3: Биллинг и автопродление подписок (Daily Billing Run)
*Запускается каждый день в 00:01.*
* **Task 1:** Поиск подписок в статусе `ACTIVE`, срок которых истекает через 3 дня. -> Отправка триггера на пре-уведомление Родителю/Взрослому.
* **Task 2:** Поиск подписок, истекающих *сегодня*. -> Отправка HTTP-запроса в Node.js Billing Service на инициализацию списания (через Stripe/CloudPayments).
* **Task 3:** Поиск просроченных подписок (конец Grace Period). -> Отправка API-запроса в Keycloak на заморозку аккаунта (удаление роли или перевод в группу `Suspended`).

## 3. Обновленная схема взаимодействия (Как это работает вместе)

Теперь наша система четко разделена на 3 слоя:

1.  **Identity Layer (Keycloak):** Стоит «на входе». Отвечает на вопрос *"Кто ты и что тебе можно?"*.
2.  **Orchestration Layer (Apache Airflow):** Стоит «сбоку». Задает ритм системы. Отвечает на вопрос *"Что нужно сделать именно сегодня/сейчас по расписанию?"*.
3.  **Execution Layer (Node.js Microservices + RabbitMQ):** Выполняет фактическую работу (показывает видео, проверяет тесты, обеспечивает чат, проводит платежи). Реагирует на действия пользователей в реальном времени и на команды от Airflow.

### Пример гибридного триггера (Студент не был 5 дней)
1.  **Airflow (DAG: Inactivity Check)** ночью находит студента Ивана, который не заходил 5 дней.
2.  **Airflow** отправляет HTTP POST запрос (через `SimpleHttpOperator`) на наш Node.js `Trigger Service` или напрямую кладет сообщение в RabbitMQ.
3.  **CRM Service (Node.js)** читает сообщение, находит ментора Ивана и создает ему карточку задачи *"Связаться с Иваном, риск оттока"*.
4.  Ментор утром заходит в систему (авторизуясь через **Keycloak**), видит задачу, пишет Ивану в чат (через **Communication Service**).
