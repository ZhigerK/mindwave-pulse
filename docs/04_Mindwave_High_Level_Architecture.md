# АРХИТЕКТУРА И ВЗАИМОДЕЙСТВИЕ КОМПОНЕНТОВ (HLA)
**Продукт:** Платформа Mindwave (Гибридная образовательная экосистема)
**Стек:** Node.js, Microservices, Event-Driven Architecture (EDA)
**Версия:** 1.0
**Статус:** Черновик

---

## 1. Архитектурные принципы и Технологический стек

Чтобы платформа Mindwave могла плавно переключаться между режимами (Adult/Child) и автоматически управлять нагрузкой менторов, архитектура строится на следующих принципах:

* **Event-Driven (Событийно-ориентированная модель):** Все значимые действия пользователей генерируют события (events), которые передаются через брокер сообщений. Это ядро для реализации нашей «Матрицы триггеров».
* **Изоляция данных (Database per Service):** Каждый микросервис имеет свою базу данных для обеспечения независимого масштабирования.
* **Технологический стек (Рекомендация):**
    * **Backend:** Node.js (рекомендуется фреймворк NestJS для строгой типизации и модульности, или Express/Fastify).
    * **Message Broker:** RabbitMQ или Apache Kafka (для маршрутизации триггеров).
    * **Базы данных:** PostgreSQL (реляционные данные: пользователи, биллинг), MongoDB (гибкие данные: контент уроков, логи прогресса), Redis (кэширование, сессии, стрики).
    * **API Gateway:** NGINX или Kong (для маршрутизации запросов от клиентов).

## 2. Уровень клиентов (Frontend Apps)

Вместо одного перегруженного интерфейса, система отдает разные клиентские приложения (или изолированные роуты SPA/PWA) в зависимости от роли:

1.  **Student App (SPA/PWA):** Интерфейс для учеников (Child и Adult). Плеер, тесты, чат с ментором, геймификация. (React/Vue).
2.  **Parent Dashboard:** Легкий интерфейс для родителей (графики, отчеты, биллинг).
3.  **Mentor CRM / Admin Panel:** Рабочее место ментора и методиста (React/Vue). Оптимизировано для десктопа. Включает WebSockets для чата в реальном времени.

Все клиенты общаются с бэкендом через единую точку входа — **API Gateway**.

## 3. Ядро системы: Состав микросервисов

Бизнес-модули из PRD декомпозируются на следующие независимые Node.js микросервисы.

### 3.1. IAM Service (Identity & Access Management)
* **Зона ответственности:** Регистрация, авторизация (JWT), управление ролями (RBAC).
* **Ключевая логика:** Именно здесь хранится связка `Parent_ID <-> Child_ID`. При авторизации сервис выдает токен, в котором зашита роль (Adult, Child, Parent, Mentor), что определяет права доступа во всех остальных сервисах.
* **БД:** PostgreSQL.

### 3.2. LMS & Content Service (Контентный движок)
* **Зона ответственности:** Хранение структуры курсов, модулей, видеоуроков, тестов.
* **Ключевая логика:** Отдает структуру маршрута, валидирует ответы на авто-тесты.
* **БД:** MongoDB (удобно хранить древовидную структуру курсов и JSON-конфигурации тестов).

### 3.3. Progress & Gamification Service (Движок вовлечения)
* **Зона ответственности:** Отслеживание того, на каком шаге находится студент, подсчет баллов (XP), контроль стриков (streaks), выдача ачивок.
* **Ключевая логика:** Слушает события от LMS Service (например, `TestPassed`). Обновляет прогресс-бар для Adult или начисляет баллы для Child.
* **БД:** Redis (для быстрых счетчиков и стриков) + PostgreSQL (для персистентного хранения истории прогресса).

### 3.4. Mentor CRM & Task Service (Оркестратор нагрузки)
* **Зона ответственности:** Алгоритм распределения студентов по менторам, генерация и хранение очереди задач (Task List) для менторов, контроль SLA.
* **Ключевая логика:** Не генерирует триггеры сам, а *принимает* команды от Trigger Engine. Хранит статусы задач (Открыта/В работе/Закрыта).
* **БД:** PostgreSQL.

### 3.5. Trigger & Notification Engine (Мозг системы)
* **Зона ответственности:** Реализация «Матрицы триггеров». Оценка условий (Rules Engine) и рассылка уведомлений (Email, SMS, Telegram, Push).
* **Ключевая логика:** Подписан на все топики в Message Broker. Если видит событие `UserOfflineFor3Days`, проверяет роль пользователя через IAM. Если это Adult -> шлет email. Если Child -> шлет команду в Mentor CRM («Создать задачу для ментора»).

### 3.6. Communication Service (Чат и Live-сессии)
* **Зона ответственности:** Внутренний мессенджер между Студентом и Ментором. Интеграция с Zoom/Google Meet API для создания ссылок на вебинары.
* **Технологии:** Node.js + Socket.io (для real-time чата).
* **БД:** MongoDB (хранение истории переписки).

### 3.7. Billing & Subscription Service (Коммерческий контур)
* **Зона ответственности:** Интеграция с платежными шлюзами (Stripe, CloudPayments), управление статусами подписок (Active, Past_Due, Canceled), контроль Grace Period.
* **Ключевая логика:** Раз в день проверяет даты окончания подписок. Генерирует события `SubscriptionRenewed` или `PaymentFailed`.
* **БД:** PostgreSQL (высокие требования к транзакционности).

## 4. Схема взаимодействия компонентов (Как это работает на практике)

Платформа использует два типа связи:
1.  **Синхронная (REST API / GraphQL):** Запросы клиента на получение данных (например, загрузить список уроков, отправить сообщение в чат).
2.  **Асинхронная (Message Broker / RabbitMQ):** Взаимодействие сервисов между собой для отработки бизнес-логики без блокировки потоков.

### Пример 1: Ученик (Child) завалил тест 2 раза (Критический путь из Матрицы триггеров)

1.  **Студент** отправляет ответы теста на API Gateway.
2.  **API Gateway** маршрутизирует запрос в **LMS Service**.
3.  **LMS Service** проверяет ответы, видит вторую ошибку. Возвращает клиенту статус `Failed`.
4.  *Асинхронно:* **LMS Service** публикует в RabbitMQ событие: `TestFailed { userId: 123, lessonId: 45, attempt: 2 }`.
5.  **Trigger Engine** читает событие. Проверяет правила: «Если attempt == 2 -> Нужна помощь ментора». Публикует команду `CreateMentorTask`.
6.  **Mentor CRM Service** читает команду, находит ментора, привязанного к `userId: 123`, и создает в его базе задачу с красным приоритетом. Через WebSockets пушит уведомление на фронтенд ментора.

### Пример 2: Формирование еженедельного отчета для Родителя

1.  *CRON Job* внутри **Trigger Engine** срабатывает в воскресенье вечером.
2.  **Trigger Engine** запрашивает у **IAM Service** список всех пользователей с ролью `Child`.
3.  Для каждого ребенка запрашивает данные за неделю у **Progress Service** (сколько XP получено) и **LMS Service** (какие уроки пройдены).
4.  **Trigger Engine** компилирует эти данные в красивый JSON.
5.  Ищет в IAM Service `Parent_ID`, привязанный к этому ребенку.
6.  Отправляет JSON в систему шаблонизации (Email/Telegram) и пушит в **Parent Dashboard**.

## 5. Преимущества этой архитектуры для Mindwave

1.  **Устойчивость к падениям:** Если ляжет Billing Service, дети продолжат смотреть уроки и сдавать тесты (Progress Service и LMS независимы).
2.  **Экономия на масштабировании:** По мере роста базы студентов, мы можем разворачивать новые инстансы только нагруженных сервисов (LMS и Progress), не трогая CRM или Биллинг.
3.  **Идеальное ложится на Node.js:** Асинхронная природа Node.js отлично подходит для Event-Driven архитектуры, работы с брокерами сообщений и WebSockets.
